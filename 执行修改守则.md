# 执行修改守则

> 本文档记录项目开发过程中的执行守则与检查清单，确保代码修改的规范性与一致性。

---

## 守则一：修改后检查未导入的包

**在完成任何 Flutter 代码修改后，必须检查修改后的代码文件是否使用了当前文件未导入的包。**

### 为什么需要这条守则

Dart/Flutter 的导入机制要求每个文件显式声明其依赖的包。在修改代码时（尤其是跨文件复制粘贴代码、重构方法、或新增 API 调用时），很容易引入对未导入包的引用，导致编译失败。

**典型场景：**
- 从文件 A 复制一段使用了 `embyApiServiceProvider` 的代码到文件 B，但文件 B 没有导入 `emby_api_service.dart`
- 重构时将某个方法移动到新的 Provider/Service 中，但忘记迁移对应的 `import` 语句
- 新增 API 调用时直接使用了类型（如 `DioException`、`BaseItemDto`），但没有导入定义该类型的包

### 检查方法

修改完成后，运行以下命令检查目标文件：

```bash
# 检查单个文件
flutter analyze lib/features/player/player_viewmodel.dart

# 检查整个目录
flutter analyze lib/features/home/

# 检查整个项目
flutter analyze
```

重点关注 `error` 级别的诊断信息，尤其是：
- `Undefined name 'xxx'` — 使用了未导入的类型/变量/方法
- `The getter 'xxx' isn't defined` — 使用了未导入的 getter
- `The method 'xxx' isn't defined` — 调用了未导入的方法

### 修复步骤

1. 找到未定义标识符的原始定义位置
2. 在文件顶部添加对应的 `import` 语句
3. 重新运行 `flutter analyze` 验证

**示例：**

```dart
// 错误：使用了 embyApiServiceProvider 但没有导入
final api = ref.read(embyApiServiceProvider);
//        ^^^^^^^^^^^^^^^^^^^^^^^
// Error: The getter 'embyApiServiceProvider' isn't defined

// 修复：添加导入
import 'package:emby_client/core/api/emby_api_service.dart';
```

### 检查清单

- [ ] 修改完成后运行 `flutter analyze` 检查目标文件
- [ ] 确认没有 `Undefined name` 或 `isn't defined` 错误
- [ ] 确认没有新增 `unused_import` 警告（导入但未使用）
- [ ] 如果修改涉及多个文件，检查所有被修改的文件
